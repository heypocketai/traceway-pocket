# OpenTelemetry Integration

Send traces and metrics from any OpenTelemetry-instrumented application to Traceway — no vendor SDK required.

[OpenTelemetry](https://opentelemetry.io/) (OTel) is the industry-standard, vendor-neutral framework for collecting telemetry. If your app already uses OTel, you can point it at Traceway with a few lines of config. If you're starting fresh, any OTel SDK will work.

## How It Works

1. **Instrument your app** with an OpenTelemetry SDK (or auto-instrumentation).
2. **Export via OTLP/HTTP** — configure the SDK (or an OTel Collector) to send data to your Traceway instance.
3. **Traceway maps the data** — spans become endpoints and traces; metrics appear on your dashboards.

## Supported Languages

Any language with an OTel SDK can export to Traceway. Here are the most common ones:

| Language | OTel SDK | Install |
|----------|----------|---------|
| Java | [opentelemetry-java](https://github.com/open-telemetry/opentelemetry-java) | [Agent JAR](https://opentelemetry.io/docs/zero-code/java/agent/) |
| Python | [opentelemetry-python](https://github.com/open-telemetry/opentelemetry-python) | `pip install opentelemetry-sdk` |
| C# / .NET | [opentelemetry-dotnet](https://github.com/open-telemetry/opentelemetry-dotnet) | `dotnet add package OpenTelemetry` |
| Go | [opentelemetry-go](https://github.com/open-telemetry/opentelemetry-go) | `go get go.opentelemetry.io/otel` |
| Node.js | [opentelemetry-js](https://github.com/open-telemetry/opentelemetry-js) | `npm install @opentelemetry/sdk-node` |
| PHP / Symfony | [opentelemetry-php](https://github.com/open-telemetry/opentelemetry-php) | `composer require open-telemetry/sdk` — see [Symfony guide](/client/symfony) |
| Cloudflare Workers | [workers-sdk](https://developers.cloudflare.com/workers/observability/otel/) | Built-in — see [Cloudflare guide](/client/otel-cloudflare) |

## Configuration

| Setting | Value |
|---------|-------|
| **Endpoint** | `https://<your-instance>/api/otel` |
| **Traces path** | `/v1/traces` |
| **Metrics path** | `/v1/metrics` |
| **Auth header** | `Authorization: Bearer <project_token>` |
| **Protocol** | OTLP/HTTP (Protobuf or JSON) |
| **Compression** | Gzip supported (`Content-Encoding: gzip`) |
| **Max body size** | 10 MB |

## Quick Start: Direct SDK Export

The simplest setup — your app exports directly to Traceway with no extra infrastructure.

Here's a Node.js example:

```typescript
import { OTLPTraceExporter } from "@opentelemetry/exporter-trace-otlp-http";
import { OTLPMetricExporter } from "@opentelemetry/exporter-metrics-otlp-http";

const traceExporter = new OTLPTraceExporter({
  url: "https://your-traceway-instance.com/api/otel/v1/traces",
  headers: {
    Authorization: "Bearer your-project-token",
  },
});

const metricExporter = new OTLPMetricExporter({
  url: "https://your-traceway-instance.com/api/otel/v1/metrics",
  headers: {
    Authorization: "Bearer your-project-token",
  },
});
```

The same pattern applies to any language — set the OTLP/HTTP endpoint and the `Authorization` header in your exporter config.

## Quick Start: OTel Collector

If you already run an [OpenTelemetry Collector](https://opentelemetry.io/docs/collector/) or want a central pipeline that fans out to multiple backends, you can route data through it. **This is optional** — the direct SDK export above works without a Collector.

```yaml
exporters:
  otlphttp:
    endpoint: "https://your-traceway-instance.com/api/otel"
    headers:
      Authorization: "Bearer your-project-token"

service:
  pipelines:
    traces:
      exporters: [otlphttp]
    metrics:
      exporters: [otlphttp]
```
